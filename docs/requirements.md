# SQL Processor 필수 구현 요구사항 정의서

## 1. 문서 목적

이 문서는 `local/수요코딩회.txt`를 기준으로 SQL Processor의 필수 구현 범위를 정의한다.

본 문서는 과제 원문에서 직접 요구한 항목만 우선 정리한 최소 요구사항 문서다.

## 2. 요구사항 근거

- 근거 문서: `local/수요코딩회.txt`
- 문서 작성 기준일: 2026-04-08

## 3. 필수 목표

필수 구현 목표는 다음과 같다.

- 텍스트 파일로 작성된 SQL 문을 CLI를 통해 전달할 수 있어야 한다.
- 입력 SQL을 파싱하여 내부적으로 구조화할 수 있어야 한다.
- 최소한 `INSERT`, `SELECT`를 지원해야 한다.
- `INSERT`는 데이터를 파일에 저장해야 한다.
- `SELECT`는 파일에서 데이터를 읽어 출력해야 한다.
- 각 테이블은 파일로 관리해야 한다.

## 4. 구현 범위

### 4.1 포함 범위

- C 언어 기반 CLI 프로그램
- SQL 파일 입력 처리
- `INSERT` 파싱
- `SELECT` 파싱
- 파싱 결과를 표현하는 내부 자료구조
- 파일 기반 테이블 저장소
- `INSERT` 실행
- `SELECT` 실행
- 기본 오류 처리
- 단위 테스트
- 기능 테스트

### 4.2 제외 범위

아래 기능은 필수 구현 범위에 포함하지 않는다.

- `CREATE TABLE`
- `UPDATE`
- `DELETE`
- `JOIN`
- `WHERE`
- `ORDER BY`
- `GROUP BY`
- 서브쿼리
- 트랜잭션
- 인덱스
- 동시성 제어
- 권한 관리
- 네트워크 서버 기능
- 별도 스키마 메타데이터 시스템
- 타입 시스템 및 타입 검증

## 5. 전제 조건

- schema 및 table은 이미 존재한다고 가정한다.
- 최소 구현에서는 테이블의 존재를 테이블 파일의 존재로 판단한다.
- 각 테이블 파일의 첫 줄은 컬럼명 헤더라고 가정한다.
- 입력 SQL은 파일 단위로 전달된다.
- 구현 언어는 C다.

## 6. 시스템 처리 흐름

필수 구현의 표준 처리 흐름은 다음과 같다.

1. 사용자가 CLI로 SQL 파일 경로를 전달한다.
2. 프로그램이 SQL 파일을 읽는다.
3. SQL 문을 파싱하여 명령 종류와 구성 요소를 추출한다.
4. 명령 종류에 따라 파일 기반 저장소에 읽기 또는 쓰기를 수행한다.
5. 성공 결과 또는 오류 메시지를 CLI에 출력한다.

## 7. 기능 요구사항

### 7.1 CLI 요구사항

- 프로그램은 명령행에서 실행 가능해야 한다.
- 사용자는 SQL이 담긴 파일 경로를 인자로 1개 전달할 수 있어야 한다.
- 프로그램은 지정된 파일을 읽어 SQL 처리를 시작해야 한다.
- 입력 파일이 없거나 읽을 수 없으면 오류를 출력해야 한다.

예시 실행 형태:

```bash
sql_processor input.sql
```

### 7.2 SQL 입력 요구사항

- 최소 구현은 한 번의 실행에서 SQL 문 1개를 처리한다.
- SQL 문은 텍스트 파일에 저장되어 있어야 한다.
- 일반적인 공백과 줄바꿈은 허용한다.
- 문장 끝 세미콜론(`;`)은 필수다.

### 7.3 최소 지원 SQL

필수 구현에서 최소 지원 SQL은 아래 두 종류다.

#### 7.3.1 INSERT

```sql
INSERT INTO users (id, name) VALUES (1, 'jungle');
```

필수 요구:

- 테이블명을 추출할 수 있어야 한다.
- 컬럼 목록을 추출할 수 있어야 한다.
- 값 목록을 추출할 수 있어야 한다.
- 컬럼 수와 값 수가 맞지 않으면 오류 처리해야 한다.

#### 7.3.2 SELECT

```sql
SELECT id, name FROM users;
```

필수 요구:

- 테이블명을 추출할 수 있어야 한다.
- 조회 컬럼 목록을 추출할 수 있어야 한다.
- 파일에서 해당 컬럼 데이터를 읽어 출력할 수 있어야 한다.

## 8. 데이터 저장 요구사항

### 8.1 저장 방식

- 각 테이블은 파일 하나로 관리한다.
- `INSERT`는 대상 파일에 데이터를 추가한다.
- `SELECT`는 대상 파일에서 데이터를 읽는다.

### 8.2 저장 포맷

- 데이터 저장 포맷은 자유롭게 설계할 수 있다.
- 단, 필수 구현에서는 단순한 텍스트 기반 포맷을 사용해야 한다.
- 최소 구현에서는 CSV 형식을 기본 저장 포맷으로 채택한다.

### 8.3 최소 CSV 규칙

- 첫 줄은 컬럼명 헤더다.
- 이후 줄은 각 레코드다.
- 테이블명 `<table>`은 `data/<table>.csv` 파일에 매핑한다.
- `INSERT`는 새 레코드를 마지막 줄에 추가한다.
- `SELECT`는 헤더를 기준으로 요청한 컬럼을 찾아 출력한다.

예시:

```text
id,name
1,jungle
2,sql
```

## 9. 실행 요구사항

### 9.1 INSERT 실행

- 대상 테이블 파일이 존재해야 한다.
- 입력 데이터를 파일 끝에 추가해야 한다.
- 저장 성공 여부를 사용자에게 알릴 수 있어야 한다.

### 9.2 SELECT 실행

- 대상 테이블 파일이 존재해야 한다.
- 파일에서 데이터를 읽어 요청한 컬럼만 출력해야 한다.
- 출력 순서는 요청한 컬럼 순서를 따라야 한다.

## 10. 오류 처리 요구사항

- 지원하지 않는 SQL 문은 오류 처리해야 한다.
- 파싱 실패 시 오류 처리해야 한다.
- 존재하지 않는 테이블 파일 조회 또는 저장 시 오류 처리해야 한다.
- 컬럼 수와 값 수가 맞지 않으면 오류 처리해야 한다.
- 존재하지 않는 컬럼 조회 시 오류 처리해야 한다.

## 11. 테스트 요구사항

### 11.1 단위 테스트

아래 항목은 단위 테스트 대상으로 포함해야 한다.

- SQL 문장 종류 판별
- `INSERT` 파싱
- `SELECT` 파싱
- CSV 행 추가
- CSV 컬럼 조회

### 11.2 기능 테스트

아래 시나리오는 최소한 검증되어야 한다.

- SQL 파일로 `INSERT`를 실행하면 데이터가 파일에 추가된다.
- 같은 테이블에 대해 `SELECT`를 실행하면 저장된 데이터가 조회된다.
- 잘못된 SQL 입력 시 실패한다.
- 존재하지 않는 입력 파일 경로 전달 시 실패한다.
- 존재하지 않는 컬럼 조회 시 실패한다.

### 11.3 엣지 케이스

최소 구현에서도 아래 케이스를 우선 검토해야 한다.

- 빈 SQL 파일
- 공백만 포함된 SQL 파일
- 지원하지 않는 문장 타입
- 컬럼 수와 값 수가 맞지 않는 입력
- 빈 결과 집합 조회

## 12. 최소 구현 완료 기준

아래 조건을 모두 만족하면 필수 구현 완료로 판단한다.

- SQL 파일을 CLI 인자로 받아 처리할 수 있다.
- `INSERT` 문을 파싱하고 실행할 수 있다.
- `SELECT` 문을 파싱하고 실행할 수 있다.
- 각 테이블을 파일 단위로 관리할 수 있다.
- `INSERT` 시 파일에 데이터를 추가할 수 있다.
- `SELECT` 시 파일에서 데이터를 읽어 출력할 수 있다.
- 기본 오류 처리가 동작한다.
- 단위 테스트와 기능 테스트가 존재한다.

## 13. 권장사항

아래 항목은 필수 구현은 아니지만 이후 보완 대상으로 고려할 수 있다.

- 더 엄격한 SQL 문법 검증
- 상세 종료 코드 체계
- 별도 스키마 메타데이터 파일
- 타입 검증
- 출력 포맷 개선
- 테스트 자동화 강화
