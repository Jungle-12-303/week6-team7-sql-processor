# 테스트 읽는 법

## 1. 왜 테스트를 같이 봐야 하나

이 프로젝트는 코드만 보면 “어디까지 정상 동작으로 보는지”가 바로 보이지 않을 수 있습니다.

테스트를 같이 보면 아래를 빠르게 알 수 있습니다.

- 어떤 입력을 지원하는지
- 어떤 실패를 막고 싶은지
- CSV 저장 규칙이 무엇인지

## 2. 테스트 파일별 역할

### `tests/unit/test_parser.c`

Parser만 따로 검증합니다.

확인하는 것:

- `INSERT` 파싱 성공
- `SELECT` 파싱 성공
- 세미콜론 누락 실패
- 빈 SQL 실패
- 공백 SQL 실패
- 지원하지 않는 SQL 실패
- 컬럼/값 개수 불일치 실패
- 키워드 대소문자 혼합 입력 성공

이 테스트를 보면 Parser의 책임이 분명해집니다.

### `tests/unit/test_storage.c`

Storage만 따로 검증합니다.

확인하는 것:

- CSV에 행 추가
- 헤더 기준 컬럼 선택 조회
- 존재하지 않는 컬럼 조회 실패
- 테이블 파일 없음 실패
- 콤마 포함 값 거부
- 손상된 CSV 행 실패

이 테스트를 보면 Storage의 제약도 함께 이해할 수 있습니다.

### `tests/integration/test_cli.c`

CLI부터 끝까지 실제 흐름을 검증합니다.

확인하는 것:

- SQL 파일로 `INSERT` 실행
- SQL 파일로 `SELECT` 실행
- 잘못된 SQL 파일 실패
- 존재하지 않는 입력 파일 실패
- 빈 SQL 파일 실패
- 공백만 있는 SQL 파일 실패

이 테스트는 “사용자 입장에서 정말 동작하는가”를 보는 테스트입니다.

## 3. 테스트가 알려주는 현재 시스템의 제약

테스트를 보면 지금 구현이 의도적으로 단순하다는 걸 알 수 있습니다.

- CSV escaping 없음
- 값 안에 콤마 불가
- 헤더를 기준으로만 컬럼 판단
- 별도 타입 검증 없음
- SQL 한 문장만 지원

즉, 테스트는 단순히 버그를 찾는 용도만이 아니라 현재 시스템의 계약을 설명하는 문서 역할도 합니다.

## 4. 앞으로 추가해볼 만한 테스트

학습이 더 진행되면 아래 테스트도 추가해볼 수 있습니다.

- `SELECT`에서 컬럼 순서가 바뀌는 경우 더 다양한 조합
- 아주 긴 SQL 입력
- 아주 긴 CSV 행
- 빈 테이블에서의 `SELECT`
- 헤더가 없는 파일
- 중복 컬럼명이 있는 헤더

## 5. 추천 학습 순서

1. `test_parser.c`를 읽는다.
2. `src/parser.c`를 읽는다.
3. `test_storage.c`를 읽는다.
4. `src/storage.c`를 읽는다.
5. 마지막으로 `test_cli.c`와 `src/cli_runner.c`를 함께 본다.

이 순서로 보면 시스템을 아래처럼 점진적으로 이해할 수 있습니다.

- 문자열 해석
- 파일 저장
- 전체 실행 흐름
